# Xteink X4 Native Chunked Refresh Design

## Summary

Move the `xteink-x4-portrait` BinBook pipeline from firmware-converted packed
GRAY2 pages to compiler-generated SSD1677-native planes. The compiler runs on a
host machine and does the expensive work: logical-to-physical rotation, GRAY2
plane decomposition, BW plane generation, 16-row chunking, chunk compression,
and adjacent-page dirty chunk map generation.

The Xteink X4 firmware should display a precompiled BinBook with minimal CPU and
RAM: read compact metadata, decode one 16-row PackBits chunk into a bounded
buffer, stream that chunk to SSD1677 RAM, and trigger the correct refresh mode.

This design intentionally updates the BinBook 0.1 candidate behavior for the
first target profile. Existing single-plane X4 BinBooks generated by the current
repo are pre-final artifacts and do not need compatibility support.

## Current Context

`BINBOOK_FORMAT_SPEC.md` already reserves e-paper plane slots:

- slot 0: MSB plane
- slot 1: LSB plane
- slot 2: BW plane
- slot 3: future delta plane

The current Python writer does not yet follow that model for X4 output. It
stores one PackBits-compressed packed GRAY2 blob in slot 0. The Rust firmware
then decompresses that blob twice, converts rows into SSD1677 red and black
planes on-device, and refreshes with `RefreshMode::Grayscale`.

SquidScript’s SSD1677 path is the behavioral reference for refresh policy:

- full grayscale refresh seeds or cleans up the panel;
- BW differential partial refresh uses the previous BW page as the old image and
  the current BW page as the new image;
- after a small number of fast page turns, a full grayscale refresh cleans up
  accumulated ghosting.

The BinBook compiler should absorb the work that SquidScript currently performs
at runtime where possible.

## Native X4 Page Encoding

For `xteink-x4-portrait`, `GRAY2_PACKED` pages are stored as SSD1677-native
1-bit planes in physical panel order, not as a single canonical packed GRAY2
blob.

The native plane geometry is fixed for this profile:

- physical width: `800`
- physical height: `480`
- bytes per physical row: `100`
- decompressed plane size: `48_000` bytes
- chunk rows: `16`
- decompressed chunk size: `1_600` bytes
- chunks per plane: `30`

Plane meanings:

| Slot | Bitmap | Meaning | SSD1677 target RAM |
|------|--------|---------|--------------------|
| 0 | `0x01` | GRAY2 MSB / red overlay plane | `0x26` |
| 1 | `0x02` | GRAY2 LSB / black plane | `0x24` |
| 2 | `0x04` | BW fast-refresh plane | `0x24` for current, `0x26` for previous |
| 3 | `0x08` | Reserved future delta plane | not used |

Each plane is already rotated from logical portrait coordinates to SSD1677
physical row order. Each bit uses the SSD1677-native polarity currently emitted
by the Rust display conversion helpers: white bits remain set, active pigment
bits are cleared.

The compiler must generate all three required planes for every X4 GRAY2 page and
set `plane_bitmap = 0x07`.

## Chunking And Compression

Each plane is divided into 30 independent 16-row chunks. The compiler compresses
each chunk independently with RLE PackBits. The page index `offset_plane_N` and
`size_plane_N` still describe the complete compressed blob for that plane, and a
new chunk index section describes the chunk offsets within `PAGE_DATA`.

The firmware must not need a full plane buffer. It decodes one compressed chunk
into a `1_600` byte buffer and writes those rows to the SSD1677. A two-buffer
path is allowed for operations that need both previous and current BW chunks,
but adjacent dirty maps should normally avoid on-device comparison.

PackBits remains the only required compression method for this design.

## New Metadata Sections

Add two binary sections. The numeric IDs fit the existing page/navigation section
range and do not conflict with current constants.

### `PAGE_CHUNK_INDEX` Section

Section ID: `44`

Record size: `24` bytes

One record exists for every `(page, plane, chunk)` tuple present in the file.
Records are sorted by page number, then plane slot, then chunk index.

```c
struct PageChunkIndexEntry {
    u32 page_number;
    u8  plane_slot;
    u8  chunk_index;
    u16 row_start;
    u16 row_count;
    u16 reserved0;
    u32 page_data_offset;
    u32 compressed_size;
    u32 uncompressed_size;
};
```

Required values for X4 full-page chunks:

- `plane_slot` is `0`, `1`, or `2`.
- `chunk_index` is `0..29`.
- `row_start = chunk_index * 16`.
- `row_count = 16`.
- `uncompressed_size = 1600`.
- `page_data_offset` is relative to `header.page_data_offset`, matching plane
  directory offset semantics.

### `PAGE_TRANSITION_INDEX` Section

Section ID: `45`

Record size: `24` bytes

One record exists for each adjacent transition in both directions:

- `0 -> 1`, `1 -> 2`, ... `N-2 -> N-1`
- `1 -> 0`, `2 -> 1`, ... `N-1 -> N-2`

```c
struct PageTransitionIndexEntry {
    u32 from_page_number;
    u32 to_page_number;
    u32 changed_chunk_mask;
    u16 first_changed_chunk;
    u16 changed_chunk_count;
    u16 flags;
    u16 reserved0;
    u32 reserved1;
};
```

`changed_chunk_mask` uses bit `chunk_index` for the 30 X4 chunks. The compiler
sets a bit when the BW plane chunk differs between `from_page_number` and
`to_page_number`. `first_changed_chunk` and `changed_chunk_count` describe the
smallest contiguous chunk window covering all changed chunks. Firmware may use
the mask for sparse streaming or use the contiguous window for simpler SSD1677
window setup. If pages are identical, the mask is `0`, `first_changed_chunk = 0`,
and `changed_chunk_count = 0`.

`flags` is `0` for this design.

## Refresh Policy

Default X4 refresh behavior is stateful:

1. If there is no valid previous page, perform a full grayscale seed refresh.
2. If the cleanup cadence is reached, perform a full grayscale cleanup refresh.
3. Clean default fast page turn: stream the full previous BW plane to red RAM
   and the full current BW plane to black RAM for the full screen, then trigger
   partial refresh. Firmware may stream this as 16-row chunks to keep RAM
   bounded.
4. Chunk-dirty adjacent page turn: firmware may stream only transition-marked
   chunks when hardware verification has proven that the SSD1677 partial refresh
   is clean for that windowed update mode. This mode is behind an explicit opt-in
   and is not the default.

The cleanup cadence is 5 fast refreshes. This follows SquidScript's current
BinBook page-turn policy.

Full grayscale refresh:

1. Stream target page plane 0 to SSD1677 red RAM (`0x26`).
2. Stream target page plane 1 to SSD1677 black RAM (`0x24`).
3. Trigger grayscale refresh (`0x21 [0x00, 0x00]`, `0x22 [0xC7]`, `0x20`).

Chunk-dirty BW differential partial refresh:

1. Use the transition record’s changed chunks.
2. Stream previous page plane 2 chunks to SSD1677 red RAM (`0x26`) for the
   changed chunk windows.
3. Stream target page plane 2 chunks to SSD1677 black RAM (`0x24`) for the same
   changed chunk windows.
4. Trigger partial refresh (`0x21 [0x00, 0x00]`, `0x22 [0xFC]`, `0x20`).

Full-screen BW differential partial refresh:

1. Stream all previous page BW chunks to SSD1677 red RAM (`0x26`).
2. Stream all target page BW chunks to SSD1677 black RAM (`0x24`).
3. Trigger partial refresh.

If a render fails, firmware must not update the previous-page state or fast
refresh count.

## Shared Display API Direction

The current Rust firmware mostly renders BinBook pages. This change should still
create a shared refresh interface instead of baking page-turn rules directly
into `main.rs`.

`binbook-fw` should own:

- refresh policy state;
- dirty chunk mask/window types;
- page transition lookup;
- chunk streaming orchestration.

`ssd1677-driver` should stay a reusable command and streaming primitive crate.
It may gain methods for writing a chunk-aligned RAM window, but it should not
know BinBook page numbers, transition records, or cleanup cadence.

Future chrome or composed display paths can reuse the same refresh API by
submitting precomputed dirty chunk masks or full-screen differential requests.
This design does not require building a SquidScript-style compositor in this
repo now.

## Tests And Acceptance

Python acceptance:

- Writer emits `plane_bitmap = 0x07` for X4 GRAY2 pages.
- Every X4 page has 90 chunk records: 3 planes times 30 chunks.
- Native plane bytes for known small GRAY2 inputs match expected MSB, LSB, and
  BW output after rotation/polarity.
- Adjacent transition records contain correct changed chunk masks.
- `nav_probe.binbook` is regenerated with native chunked planes.

Rust reference acceptance:

- Section IDs 44 and 45 parse correctly.
- X4 GRAY2 plane uncompressed size is `48_000` bytes.
- Chunk lookup rejects out-of-bounds offsets and missing records.

Firmware host-test acceptance:

- Refresh policy chooses full seed, adjacent dirty partial, arbitrary
  full-screen differential, and cadence cleanup.
- Chunk-slice lookup finds plane/chunk data without copying a full page.
- Failed render leaves previous-page state unchanged.

Hardware acceptance:

- First render uses grayscale refresh.
- Adjacent page turns use transition chunk metadata.
- Arbitrary non-adjacent jump uses full-screen BW differential partial refresh.
- Every fifth fast refresh performs grayscale cleanup.
- Verification evidence comes from debug logs or serial output behind a
  compile-time feature.
